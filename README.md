# Rhythm Parry

> 1ボタンの音ゲー。飛来する斬撃をビートに合わせて弾く。成功すれば作曲済みの旋律が鳴り、ミスすればその音が消えて旋律に穴が空く。
> A one-button rhythm game built in [Pyxel](https://github.com/kitao/pyxel). Parry the incoming slashes on the beat — nail them and you *perform* the melody; miss and that note drops out.

128×128ピクセル / 16色 / 4音チャンネルだけで作った、依存ゼロ・単一ファイルのゲーム。スマホはタップだけ、30秒ノーフェイル、**交通機関の待ち時間の暇つぶし**に特化しています。

**🎮 ブラウザで遊ぶ:** https://tmgv3602.github.io/rhythm-parry/

---

## 遊び方

```bash
pip install pyxel
python rhythm_parry.py
```

| キー / タップ | 動作 |
|---|---|
| 画面タップ / `SPACE` / `Z` / `X` | パリィ |
| タップ / `R` | リトライ |

**スマホ対応**: `docs/index.html` をGitHub Pagesで配信すれば、スマホのブラウザで**画面タップだけで**遊べます。インストール不要。

**LINE等で共有**: GitHub Pagesのページにはタイトル・サムネイル(`docs/ogp.png`)のOGP情報が入っているので、URLを貼るとプレビューカードが表示されます。受け取った相手は、LINEのアプリ内ブラウザだと音やWASMが不安定なことがあるため、メニューから**外部ブラウザ(Safari/Chrome)で開く**と確実です。

**30秒・ノーフェイルのタイムアタック。** キック/スネアのバックビートとベース進行が鳴り続けるので、**斬撃が画面中央の基準線に届く瞬間=ビートに合わせて**タップする。タイミングが合うとその音がパルス波で鳴り、連続成功で `c3 e3 g3 a3 …` というアーチ状の旋律が完成する。ミスしても止まらず、コンボが切れる(=その音が鳴らず旋律に穴が空く)だけ。毎回きっちり30秒走り切るので、隙間時間にサッと開いて遊べる。

## このゲームが問うていること — 制約駆動の設計記録

これは「小さいゲーム」であると同時に、**「ゲームの本質はどこにあるか」を制約で炙り出す実験**でもある。開発の過程で見えてきたことを記録しておく。

### 1. パリィの本質は「2つの時間窓の重なり」

パリィの気持ちよさは、突き詰めると1行に還元できる。

```python
diff = abs(slash.hit_frame - beat_frame)   # 斬撃の到達時刻と入力時刻の差
if diff <= window:                          # 窓内なら成立
    perfect = diff <= PERFECT_LATE          # ±2フレームならPERFECT
```

斬撃が判定面を通過する瞬間と、プレイヤーの入力。この2つの時間窓が重なるか否か——それだけ。Sekiroの弾きも Metal Gear Rising のパリィも Beat Saber も、次元と入力デバイスは違えど判定の数式は同型になる。

### 2. 位置は「積分」せず「時間の純粋関数」にする

斬撃の位置は毎フレーム `x += speed` で積分**していない**。

```python
def x_at(self, beat_frame):
    return PARRY_POINT - (self.hit_frame - beat_frame) * SLASH_SPEED
```

位置を時間の純粋関数として宣言することで、浮動小数の累積誤差が出ず、`beat_frame == hit_frame` のとき `x` が厳密に `PARRY_POINT` になる。「ビート上にぴったり着弾する」保証が成立し、音ゲーとして破綻しない。これが本作で一番効いている設計判断。

### 3. ヒットストップを捨てた

初期版にはヒットストップ(成功時に時間が止まる演出)があった。リズムゲー化にあたって**あえて削除**した。時間を止めるとメトロノームのテンポが崩れるから。Beat Saberにヒットストップが無いのは技術的制約ではなく必然で、リズムゲーの快感は「時間が止まること」ではなく「打撃がビートに乗ること」から来る。

### 4. 譜面はデータ、エンジンは汎用

旋律・ベース・コードは宣言的データとして分離されている。

```python
MELODY = ["c3", ".", "e3", ".", "g3", ".", "a3", "g3", ...]  # 各8分ステップの音
BASS   = ["c1", "c1", "a1", "a1", "f1", "f1", "g1", "g1", ...] # コード進行の根音
ARP    = {"c3": "c3e3g3", "e3": "e3g3c4", ...}                # 単音→擬似和音
```

`MELODY` の1行を書き換えれば別の曲を演奏できる。Strudel/TidalCyclesのパターン記法と同じ発想で、エンジン(判定・生成・描画・音響)はそのまま。

### 5. ファミコン2A03の音源構成をそのまま写す

4チャンネルを役割で固定割当している。

| ch | 役割 | 音色 |
|---|---|---|
| 1 | リード旋律(プレイヤーが演奏) | パルス/矩形波 |
| 2 | ベース | 三角波 |
| 3 | ドラム(キック/スネア/ハット) | ノイズ |
| 0 | 効果音 | ノイズ |

同時発音数が足りないハードで和音を出すために、`ドミソ` を超高速で順番に鳴らす「アルペジオ奏法」もファミコン由来の技。

### 6. 罰しない — ノーフェイルという選択

開発の途中で「交通機関の待ち時間の暇つぶし」という用途に絞った。そこで気づいたのは、**遅延でイライラしている人を `GAME OVER` で追い打ちするのは逆効果**だということ。初期版にあったHP・被弾死を撤廃し、毎回必ず30秒走り切るノーフェイルにした。ミスの罰はコンボが切れる(=旋律に穴が空く)だけ。発散を目的にするなら、ゲームは「罰する」より「受け止める」方がいい。セッション長が一定になることで「サッと開いてサッと終わる」予測可能性も生まれ、隙間時間との相性が増した。

## カスタマイズ

ソース冒頭の定数とデータを変えるだけで挙動が変わる:

- `BPM` — テンポ。斬撃速度は自動で追従する
- `MELODY` / `BASS` — 譜面そのもの。自分で作曲できる
- `ACTIVE_FRAMES` — 判定窓(±フレーム)。小さいほどシビア(既定7、ソースで調整)
- `TIME_LIMIT_SEC` — セッションの長さ(既定30秒)

## アーキテクチャ

`pyxel.run(update, draw)` の契約の上に、**クロック駆動・データ解釈型のシーケンサー**が乗っている。

- `beat_frame` が単一の時計。ここからシーケンサー発火・斬撃位置・判定が全て導出される
- 2つのドライバー(時間 `_on_step` / 入力 `_read_parry`)が `Game` 状態に書き込み、`draw` はそれを純粋に読むだけ
- 演出(フラッシュ・シェイク・火花・ビートパルス)は全て「毎フレーム減算される整数タイマー」という統一パターン
